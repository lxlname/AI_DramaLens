# YAML Schema 定义与设计说明

## 1. 设计概述
本项目旨在将非结构化的小说长文本，通过大语言模型自动转换为结构化的剧本初稿。为了满足题目中“让作者可以快速获得可编辑、可进一步打磨的剧本初稿”的核心需求，本系统放弃了传统的纯文本段落输出，采用 YAML 数据序列化格式对剧本进行原子化拆解。
考虑到影视工业中“电视剧（重台词与时间轴）”与“电影（重镜头与视听）”的制作标准存在本质差异，本系统在底层设计了两套独立的 YAML Schema。

## 2. 电视剧本（文学剧本）Schema 定义

### 2.1 数据结构
```yaml
- 章节标题: string
  元数据: 
    类型: string
    基调: string
  场景列表:
    - 场景编号: integer
      场景标题: string
      场景时间: string
      环境氛围: string
      人物出场: array (string)
      剧本正文:
        - 动作: string
        - 台词: 
            角色: string
            语气: string (可选)
            内容: string
        - OS: 
            角色: string
            内容: string
```

### 2.2 设计原因与优势
1.时间轴序列化：
传统的 AI 剧本生成往往将一个场景的动作集中写在一段，台词集中写在另一段，导致音画脱节。本 Schema 在 剧本正文 节点下采用 List（数组）结构，强制要求数据以 - 动作、- 台词、- OS 的顺序交替出现。这严格还原了人类编剧“走位-说话-反应”的线性创作逻辑。

2.台词对象的原子化：
将台词拆分为 角色、语气 和 内容 三个独立字段。作者在进行二次打磨时，可以精准定位并修改某一角色的特定情绪（语气），而不会破坏整体文本结构。该结构也为后续接入 TTS（文本转语音）引擎进行自动化配音预留了数据接口。

3.保留心理描写（OS）：
电视剧本允许通过画外音或内心独白展现人物弧光。独立设置 OS 节点，能够有效承载原著小说中丰富的心理描写，避免信息丢失。

## 3. 电影剧本（分镜剧本）Schema 定义

### 3.1 数据结构
```yaml
- 章节标题: string
  元数据: 
    类型: string
    基调: string
  场景列表:
    - 场景编号: integer
      场景标题: string
      环境氛围: string
      节奏要求: string
      出场角色: array (string)
      分镜序列:
        - 镜头规格: string (如：全景/中景/近景/特写)
          运镜指令: string (如：推/拉/摇/移/固定)
          动作描写: string
          对话内容: string
          音效提示: string
          光影指令: string
```

### 3.2 设计原因与优势
1.视听维度的强制拆解：
电影剧本的核心是“画面”。本 Schema 移除了文学性的“剧本正文”，代之以 分镜序列。每一个 List Item 代表一个独立的物理镜头。通过强制拆分 镜头规格、运镜指令、音效提示 和 光影指令，使生成的初稿不仅服务于编剧，更可直接作为导演和摄影指导的分镜参考底稿。

2.动作描写的“去心理化”约束：
电影镜头无法直接拍摄人物的内心活动。因此，本 Schema 移除了 OS 节点，并在系统 Prompt 中严格约束 动作描写 字段仅能包含可见的物理动作与微表情。这倒逼 AI 将小说中的抽象心理描写转化为具象的视觉动作，提升了剧本的专业度。

3.节奏控制节点：
在场景层级增加 节奏要求 字段，便于创作者在宏观上把控单场戏的剪辑速率（如：凌厉、舒缓），为后期的动作比重分配提供依据。

## 4. 总结
本系统设计的 YAML Schema 严格遵循了“数据与表现分离”的软件工程原则。通过将复杂的剧本元素原子化、结构化，不仅大幅降低了小说改编的门槛，更确保了输出的初稿具备高度的机器可读性与人工可编辑性，完全契合现代影视工业的数字化生产流程。
